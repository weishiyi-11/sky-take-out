1.现在的密码加密方式采用的是md5的加密方式，存在以下优缺点：

    MD5 的优缺点
    优点：速度极快
    计算开销小，加密效率高，老项目、简单场景跑起来很轻量。
    使用简单、普及度极高
    几乎所有语言都内置支持，一行代码就能用，兼容性无敌。
    定长输出
    无论原字符串多长，结果都是 32 位十六进制字符串，存储方便。
    单向不可逆
    理论上不能直接解密，比明文存储稍微 “像点加密”。
    缺点（致命，不适合存密码）：
    安全性极低，极易被破解
    彩虹表、暴力破解非常成熟，随便一个在线网站就能秒解常见密码。
    碰撞率高
    不同内容可以生成相同 MD5 值，早已被证明不安全。
    不加盐等于裸奔
    相同密码永远生成相同 MD5，库泄露后批量破解极快。
    算法设计老旧，抗攻击能力差
    不抗 GPU / 字典攻击，现在只能用来做文件校验，不能存密码。
    不符合安全规范
    等保、企业安全规范明确禁止 MD5 存储用户密码。
后续更换为BCrypt加密方式，存在以下优缺点：

    优点：
    自带随机盐，不用自己处理
    每次加密同一个密码，生成的结果都不一样，从根本上防彩虹表、防批量破解。
    慢哈希算法，抗暴力破解极强
    故意设计得计算慢，GPU / 字典破解成本极高，是密码存储的理想特性。
    安全性高，行业主流标准
    Spring Security 默认推荐，企业级项目几乎都用它，过等保没问题。
    使用极其简单
    一行加密、一行校验，不用管盐存哪、怎么拼接，框架全包了。
    可调整强度
    可以通过 cost 参数调慢加密速度，硬件变快了你就把强度调高，长期安全。
    结果自带格式信息
    加密串里已经包含盐、版本、强度，直接存数据库就行，不用额外字段。
    缺点：
    性能消耗比 MD5 大很多
    因为故意慢，高并发场景下会比 MD5 更耗 CPU。
    但登录接口本来 QPS 不高，正常项目完全无感。
    密文长度固定偏长
    BCrypt 结果长度约 60 字符，数据库字段要预留够长度。
    无法自己控制盐
    盐是自动生成的，如果你有特殊需求要自定义盐，BCrypt 不太方便。
    不适合加密大量数据
    只适合密码这种短文本，不适合文件、大文本加密。
    算法较老，理论安全性不如 Argon2
    但依然足够安全，只是不是 “天花板级别”。
